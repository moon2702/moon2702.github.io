Redis 数据结构

发表于2024-07-24|学习经历

什么是Redis

Redis是一个开源项目，是使用C语言编写，基于内存的一个高性能 键值对存储系统。通常被用于缓存、消息队列、事件的发布订阅和分布式锁 。支持多种数据类型，基本数据类型有5种：String、List、Set、Zset、Hash。

Redis 特点

Redis还支持多种机制，比如事务、持久化、集群方式(主从复制、哨兵模式和切片集群模式)、发布订阅、内存淘汰机制、过期删除机制等等。

  • 高性能：Redis 存储基于内存，这意味着Redis 的读写速度非常快，Redis通常采用单线程 和异步IO 来实现读写操作，避免了多线程的竞争和阻塞，做到了高性能的读写。每秒11万次读操作，8万次的写操作。
  • 多种数据类型：Redis 除了支持基本的数据类型之外，还支持Bitmaps(位图)、HyperLogLog(基数统计)、GEO(地理信息)、Stream(流)等，并且Redis 对数据的操作都是原子性的，Redis 执行命令使用单线程，不存在并发竞争的问题。Redis 的数据类型指的是 value 的数据类型，key 部分永远都是字符串类型。
  • 持久化：Redis 提供两种持久化方式，即RDB(快照) 和AOF(日志追加) ，RDB 将内存数据以二进程方式写入磁盘，AOF 通过追加方式来记录Redis 的操作命令来实现持久化。
  • 发布订阅：Redis 支持发布订阅模式，可以被用于消息代理。发布者将消息发送到指定频道，订阅者通过接收和处理这些消息。这种模式在实时通信、事件驱动系统和消息队列等场景十分有用。
  • 分布式：Redis 通过主从复制 和分片 来实现数据的分布式存储和高可用性。主从复制将数据复制在多个从节点上，可以实现读写分离和数据备份；分片指的是将数据分布在多个Redis 节点上，实现横向扩展和负载均衡。
  • 支持事务：Redis 支持事务，可以将多个操作组合成一个原子性的操作序列，保证这些操作全部成功或者全部失败。

Redis 的使用场景

Redis 由于高性能和丰富的机制，得到了广泛的使用和认可，成为许多业务中不可或缺的组成部分。在此对Redis 的使用场景做出一些概括：

热点数据缓存

缓存是Redis 最常见的应用，例如Redis 作为MySQL 的缓存，主要是由于Redis 的高性能和高并发两个特点。

  1. 高性能

MySQL 的数据存储在硬盘上，用户第一次访问某些数据时，由于磁盘的IO限制，导致访问效率很低，如果将用户访问的数据存放在Redis 中，用户获取数据直接和内存进行交互，速度很快。

不过MySQL 中的数据改变之后，需要同步修改Redis 缓存中相应的数据，会出现Redis 和MySQL 一致性的问题。

  1. 高并发

Redis 处理请求的能力，是MySQL 的10倍，因此，访问Redis 能够承载的压力远远超过访问MySQL ，所以我们将热点数据从MySQL 中拷贝到Redis 中，可以提高系统的承载能力。并且Redis 是支持事务的，在使用时可以保证数据的一致性。

限时业务

Redis 中可以使用 expire 命令来设置一个键的生存时间，在时间结束之后，Redis 会自动删除这个键，这一特点可以使得Redis 在一些验证码、限时活动等场景下有很好的适用性。

计数器

Redis 通过 incrby 命令可以实现原子性的递增操作，可以运用于高并发的活动，例如限量商品的售卖，限制IP的请求数量等。

分布式锁

分布式锁是一种在分布式系统中控制并发访问共享资源的机制。用于确保多个节点或者进程同时访问同一资源时，只允许一个节点或者进程在同一时间对资源进行操作，从而避免数据不一致和竞争。

分布式锁具有以下特点：

  • 互斥性：分布式锁确保同一时刻只有一个客户端能够持有锁，对共享资源进行操作；
  • 安全性：分布式锁可以确保只有一个客户端能够成功获取锁，并且其他客户端不能误认为锁被释放；
  • 容错性：在出现网络分区、节点故障等情况，分布式锁应该具有一定的容错能力，确保锁的可靠性和一致性；
  • 解锁机制：需要明确的锁释放机制，无论是正常释放还是超时释放。

Redis 中分布式锁的实现

  1. 通过 SETNX 和 EXPIRE 命令来实现简单的分布式锁：
    • 通过 SETNX 尝试设置一个键，如果键不存在则设置成功，表示锁获取成功；如果键已经存在则设置失败，表示锁被占用；
    • 通过 EXPIRE 来为锁设置一个过期时间，可以防止客户端在崩溃之后无法释放锁。
  2. 通过 SET 来设置一个分布式锁，可以同时设置过期时间，保证锁实现的原子性。
  3. RedLock 算法：适用于Redis 集群环境，保证多个Redis 实例间分布式锁的安全性和可靠性。
  4. 基于Zookeeper 的分布式锁：Zookeeper 是一种分布式协调服务，提供了高可用、高性能的数据一致性服务。通过节点(znode)机制实现。
    • 临时顺序节点：客户端创建一个临时顺序节点，Zookeeper 为每个节点分配一个唯一的序列号，客户端通过获取最小的序列号来获得锁。
    • 监听机制：如果客户端未获得锁，它将会监听比它序列号小的节点的删除事件，一旦该节点被删除，客户端检查自己是否可以获取该锁。
字段 作用

分布式锁的使用场景

  1. 分布式事务：确保多个操作的原子性。
  2. 资源协调：控制共享资源的并发访问，例如数据库记录、文件等。
  3. 任务调度：在分布式任务调度系统中，确保任务不被重复执行或者访问。

集合运算

Redis 中利用Set 结构，取多个集合的交集、并集或者差集。

例如，获取两个用户的共同关注等功能。

消息队列

Redis 中可以使用List 结构来实现简单的消息队列，也可以通过发布/订阅(Pub/Sub)机制，来实现消息分发。

  • List：使用 LPUSH 和 RPOP 可以实现简单的消息队列；
  • Pub/Sub：使用 PUBLISH 和 SUBSCRIBE 可以实现实时消息分发。

Redis 数据结构

Redis 有五种基本的数据类型，String、List、Set、Zset、Hash。本篇文章仅仅介绍这五种基本数据类型。

在接触Redis 的数据结构之前，需要先了解redisObject 的概念。

redisObject

在Redis 中，无论是什么数据类型，都是通过redisObject 对象来进行存储的，redisObject 对象的定义如下：

type

type 字段表示对象的类型，占用4个bit；包含REDIS_STRING（字符串）、REDIS_LIST（列表）、REDIS_HASH（哈希）、REDIS_SET（集合）、REDIS_ZSET（有序集合）。

我们可以执行type 命令来获取对象的类型 type '键名'。

encoding

encoding 字段表示对象内部编码，占用4个bit；例如对于字符串，包含int、embstr、raw 三种编码。

可以通过执行命令来获取对象的编码 object encoding '键名'。

lru

lru 字段记录对象最后一次被命令程序访问的时间，占用大小根据版本有所不同，例如4.0版本占用24bit。

通过对比lru时间和当前时间，可以计算出对象的空转时间（单位为秒）；可以通过object idletime来显示空转时间，该命令不会改变对象的lru值 object idletime '键名'

lru 字段还与内存回收机制相关，如果Redis 打开了maxmemory 选项，并且内存回收算法选择的是volatile-lru或者allkeys-lru，那么当Redis内存占用超过maxmemory指定的值时，Redis 会优先选择空转时间最长的对象进行释放。

refcount

refcount 字段记录对象被引用的次数，占用4个字节的空间；用于对象的引用计数和内存回收。

当创建新对象时，refcount 初始化为1；当有新程序使用该对象时，refcount+1；当对象被一个程序释放时，refcount-1；当refcount 变为0时，系统释放该对象。

Redis 中被多次使用的对象被称为共享对象，共享对象目前只支持整数值的字符串对象。Redis 为了节省内存，当出现一些重复对象值时，新的程序不会创建新的对象，而是使用系统中已经存在的对象。这个被重复使用的对象就叫共享对象。

共享对象的引用次数可以通过命令来查看 object refcount '键名'，通过命令，证明了只有 0~9999 的整数值才能使用共享对象。

ptr

ptr 字段是一个指针变量，存放具体数据的地址，占用空间和系统有关。

五种数据结构的具体实现

Redis 不同的数据类型有很多底层结构实现，下面是两个版本的Redis 数据类型和底层结构的关系图。

String

String 是Redis 中最基本的数据类型，可以存储任意类型的数据，例如文本、数字、图片或者序列化的对象。一个String 类型的键最多存储 512MB 大小的数据。

String 底层实现基于SDS(simple dynamic string) ，SDS 指的是简单动态字符串，是Redis 自己实现的一种字符串结构。

SDS 相较于C语言 的字符串：

  • SDS 可以保存文本数据，还能保存二进制数据：因为SDS 使用len 而不是空字符来判断字符串是否结束，并且SDS 中所有API 都会以处理二进制的方式来处理SDS 存放在buf[] 数组中的数据，所以SDS 不仅能存放文本数据，还能存放图片、视频、音频、压缩文件等二进制数据。
  • SDS 获取字符串长度的时间复杂的是O(1)：C语言 中字符串并不会记录自身长度，需要通过遍历来获取自身长度，时间复杂度为O(n)；而SDS使用len 来记录字符串长度，复杂度为O(1)。
  • SDS API 是安全的，拼接字符串不会造成缓冲区溢出：SDS在拼接之前，先检查SDS 剩余空间是否满足需求，如果空间不够拼接，会自动扩容，不会导致缓冲区溢出。

实际上，SDS 为了兼容C语言的字符串类型，仍然会在字符串结尾添加 “\0”

内部编码

SDS有三种编码类型：

  • int：存储整数类型
  • embstr：占用64Bytes的空间，存储44Bytes的数据
  • raw：存储大于44Bytes的数据

embstr 结构存储小于等于44个Byte的字符串，例如存储44Byte大小的信息，需要开辟64个Byte大小的空间用于创建embstr，其他19个Byte大小的空间被用来存储redisObject 和SDS 的其它信息。

raw 结构存储大于44个Byte的字符串。

实际上，embstr存储数据的长度在不同Redis版本中，是不一样的：

  • 2.x：32字节
  • 3.x~4.x：39字节
  • 5.x：44字节


embstr 和raw 都使用SDS 来保存数据，它们不同之处在于：

  • 创建过程：embstr 申请一块连续的内存来保存redisObject 和SDS，raw 会申请两块连续内存分别保存redisObject 和SDS。
  • 可修改性：embstr 是不可以修改的，如果字符串长度增加，需要重新分配内存时，整个redisObject 和SDS 都需要重新分配空间，所以embstr 对象实际上是只读的，需要修改embstr 对象时，会直接将对象的编码修改为raw，然后执行修改操作。

编码转换

  • 当int 类型数据不再是整数，或者大小超过了long 的范围时，自动转换为raw 编码；
  • 当embstr 对象进行修改时，无论是否超过长度限制，都会先转换为raw 编码，然后再进行修改。

List

List 用来表示一个有序的字符串列表，按照插入顺序排序，支持在两端插入或者删除元素，一个List类型的键最多可以存储 $2^{32} - 1$ 个元素。

List 的底层数据结构根据Redis 的版本不同有所变化：

  • Redis 3.2之前，List 使用 linkedlist(双向链表) 和 ziplist(压缩列表)；
  • Redis 3.2~7.0之间，List 使用 quicklist，实际上相当于 linkedlist(双向链表) 和 ziplist(压缩列表)的结合；
  • Redis 7.0之后，List 使用 quicklist，相当于 linkedlist(双向链表) 和 listpack(紧凑列表)的结合。

内部实现：底层数据结构的选择？

  • 如果List 中的元素个数小于512个，存储的值都小于64字节时，Redis 使用 ziplist 作为List 的底层数据结构；
  • 如果List 中元素不符合上面的条件，Redis 将使用 linkedlist 来作为List 的底层数据结构；
  • 在Redis 3.2之后，直接使用 quicklist 来作为List 的底层数据结构。

底层数据结构转换的阈值可以通过修改配置来进行设置：

  • list-max-ziplist-entries： 用于配置元素个数；
  • list-max-ziplist-value： 用于配置元素值的大小。

底层实现

ziplist

压缩列表是一系列的连续内存块组成的顺序结构，整体结构如下：

ziplist 变长编码的含义

  • previous_entry_length：如果长度为1，表示上一个节点占用空间小于254字节；如果长度为5，则表示长度超过254字节。

    为什么不能表示255呢？

    • 这是由于zlend 的值就是0xFF，如果长度能表示255，将会产生歧义。
  • encoding：如果最高位为11，表示数据类型为整数；如果最高位不是11，表示数据类型是字符串
    • 00：表示encoding占用1字节，可以表示小于$2^{6} - 1$字节的字符串
    • 01：表示encoding占用2字节，可以表示小于$2^{14} - 1$字节的数据
    • 10：表示encoding占用5字节，可以表示小于$2^{32} - 1$字节的数据
    • 11000000：整数均占用1字节，表示数据部分存储2字节的int16_t类型整数
    • 11010000：表示数据部分存储4字节的int32_t类型整数
    • 11100000：表示数据部分存储8字节的int64_t类型整数
    • 11110000：表示数据部分存储3字节长度的整数
    • 11111110：表示数据部分存储1字节长度的整数
    • 1111xxxx：encoding 和content 部分合二为一，xxxx表示数据部分，可以表示0001~1101之间13个数值，分别用来表示0~12（0000、1110、1111均和其他编码冲突）

quicklist

quicklist 也是一个双向链表，并且它的每个节点都是一个 ziplist，ziplist 本身是一个能够维持数据先后顺序的列表，并且内存紧凑。quicklist 中的数据分布是这样的：例如，一个有3个节点的 quicklist，每个节点 ziplist 中包含4个数据项，那么这个 quicklist 对外体现出的结果就是包含了12个数据项目的链表。

在Redis 7.0之后 ziplist 被替换为 listpack。

quicklist 这样设计有什么好处呢？

  • 双向链表便于在表的两边进行push 和pop 操作，但是双向链表的内存开销很大，每个节点除了保存数据之外，还需要保存两个指针；
  • 双向链表各个节点是单独的内存块，地址不连续，节点多容易产生内存碎片；
  • ziplist 是一块连续的内存，存储效率高，但是不利于修改，每次数据更新都会realloc，当 ziplist 特别长时，可能导致大批量的数据拷贝，降低性能。

quicklist 的设计就是将两者的优缺点进行折中处理，那么既然需要将两者结合起来使用，ziplist 长度设置多少合适呢，Redis 提供一个选项，可以根据实际情况来进行配置 list-max-ziplist-size。这个参数可以取正值，也可以取负值。

  • 取正值时，按照数据项目个数来限制 ziplist 的长度，当配置为5时，表示 ziplist 中最多包含5个数据项。
  • 取负值时，按照字节数来限制每个 ziplist 的长度，并且只能取-1~-5这5个值：
    • -5：大小不能超过64KB；-4：不能超过32KB；-3：不能超过16KB；-2：不能超过8KB；-1：不能超过4KB。

当列表很长时，中间的数据被访问的频率比较低，List 还提供了一个选项，可以将中间的数据节点进行压缩，从而进一步节省内存空间，list-compress-depth 这个参数表示一个 quicklist 两端不被压缩的节点个数（双向链表的节点），特殊的，当该选项为 0 时，表示所有节点都不压缩，所以说，quicklist 链表的头节点和尾节点都是不会被压缩的。

对于节点的压缩方式，quicklist 使用一种无损压缩算法 LZF，对访问频率低的ziplist 进行压缩处理。

Hash

Hash 是一个键值对的集合，可以存储多个字段和值，一个Hash 类型的键最多可以存储 $2^{32} - 1$ 个元素。

Hash 的底层实现有两种：

  • ziplist：压缩列表，在达到一定阈值时，会自动转换为 hashtable 结构；
  • hashtable：哈希表。

内部实现：底层数据结构如何进行选择？

  • 如果Hash 中的元素个数小于512个，存储的值都小于64字节时，Redis 使用 ziplist 来作为Hash 的底层数据结构；
  • 如果Hash 中的元素不符合上面的条件，Redis 将使用 hashtable 来作为Hash 的底层数据结构。

在Redis 7.0版本之后，ziplist 被弃置，使用 listpack 结构（紧凑列表）来代替。

底层数据结构转换的阈值可以通过修改配置来进行设置：

  • hash-max-ziplist-entries： 用于配置元素个数；
  • hash-max-ziplist-value： 用于配置元素值的大小。

底层实现

listpack

listpack 紧凑列表，相较于 ziplist 来说，内存布局、实现原理非常相似，但是 listpack 更加紧凑，实现更加简洁。

属性 大小 作用
tot-bytes 4 Byte 记录整个列表占用的字节数
num-element 2 Byte 记录列表中总的节点个数
element-x 不定 列表中具体的各个节点
listpack-end 1 Byte 结束标识符，固定为0xFF
属性名 大小 作用
encoding 1 Byte / 2 Byte 记录当前节点的编码类型
data 不定 记录节点数据本身
len 1 Byte ~ 5 Byte 记录编码类型和元素数据的长度

encoding：针对不同长度的整型和字符串定义了多种编码方式

存储数据类型：

  • UINT结尾：表示无符号整型
  • INT结尾：表示有符号整型
  • STR结尾：表示字符串类型

存储数据长度：

  • UINT 和INT 中BIT 前的数字表示数据本身所占的比特数
  • STR 中BIT 前的数字表示字符串长度所占用的比特数
  • 并且如果BIT前的数字不是2的幂次方时，表示会占用encoding 编码中的剩余可用空间

举例：

  • LP_ENCODING_7BIT_UINT 0：表示encoding 编码为0000 0000，只会使用高地址第1位来表示标识编码 0，剩下7位来存储无符号整型数据；
  • LP_ENCODING_6BIT_STR 0x80：表示encoding 编码为1000 0000，只会使用高地址的前2位来标识编码 0x80，剩下6位来存储字符串的长度信息。

len：len 用来记录element(encoding+data) 的长度，它有如下的特点；

  • len 只是记录encoding+data 的长度，并不包括len 自身的长度；
  • len 的最高位用于标识len 的结束，而不包含长度信息；
  • len 至少有1个字节，最长为5个字节len 结构图解
listpack的遍历

在ziplist 中，每个元素记录了上一个元素的大小，支持了从后向前遍历，但是却导致了连锁更新的问题。listpack解决了该问题，每个元素不在记录上一个元素的大小，而是改为记录自身的大小，这样每个元素相互独立，修改不会相互影响。

正向遍历：

  • 首先跳过6字节的首部，指针指向第一个元素；
  • 然后根据元素的encoding 字段得到元素的长度和类型；
  • 访问当前元素或者跳过当前元素，直到访问到结束标识符，遍历结束。

反向遍历：

  • 首先访问 listpack 的前面4个字节得到总长度，定位到末尾位置；
  • 然后指针左移访问最后一个元素的len 字段，得到当前元素的长度，定位到当前元素的encoding 字段；
  • 访问当前元素或者跳过当前元素（指针左移访问上一个元素的len 字段），以此类推进行反向遍历。

连锁更新指的是在由于每个元素之间相互有联系，在修改列表中某个元素时，出现连锁反应，后面的元素都会发生修改。（记录元素长度的字段是变长编码，字段长度发生改变）

连锁更新问题出现的原因：

  • 每个元素记录上一个元素的长度；
  • 在记录上一个元素长度时使用变长编码。
hashtable

Set

Set 与List 类似，都可以用来存储多个字符串，但是Set 和List 又有所不同：Set 中的元素是无序的，不能通过索引来操作元素，Set 中的元素不能重复；一个Set 类型的键可以存储 $2^{32} - 1$ 个元素，除了支持增删改查，Redis 还支持多个Set 取交集、并集和差集。

Set 的底层实现有两种：

  • intset：整数集合，实际上是一种紧凑的数组结构，只保存 int 类型的数据，将所有的元素按照从小到大的顺序存储在一块连续的内存中，会根据传入数据的大小，将数据编码分为 int16_t、int32_t、int64_t。
  • hashtable：哈希表。

内部实现：底层数据结构如何进行选择？

  • 如果Set 中的元素全部为整数，并且保存的元素数量小于512个时，使用 intset 作为Set 的底层数据结构；
  • 如果Set 中元素不满足上面条件，使用 hashtable 作为Set 的底层数据结构。

在Redis 7.2之后，Set的底层数据结构添加了 listpack。在没有使用 listpack 之前，字符串类型的数据会直接使用 hashtable 结构，而在加入 listpack 之后，字符串类型的数据会先使用 listpack 来进行存储，在达到一定阈值时，转换为 hashtable。
这是因为 hashtable 的空间开销和碰撞概率都很高，使用 listpack 提高了内存利用率和操作效率。

set-max-intset-entries：用于配置 intset 的最大元素个数；
set-max-listpack-entries：用于配置 listpack 的最大元素个数；
set-max-listpack-value：用于配置 listpack 的单个元素大小。

intset

intset 是一个有序的整数集合，其中只能存储整数，可以存储三种类型的整数，分别为2、4、8字节。

属性 大小 作用
encoding 4 Byte 数据编码，用于表示每个元素占用的大小
length 4 Byte 元素个数，记录intset中的元素数量
contents 不定 柔性数组，用于存储实际的数据

encoding：有三种取值

  • INTSET_ENC_INT16：表示每个元素占用2字节
  • INTSET_ENC_INT32：表示每个元素占用4字节
  • INTSET_ENC_INT64：表示每个元素占用8字节

Zset

Zset 是一个有序的集合类型，用于存储不重复的字符串元素。Zset 中会给每个元素关联一个double 类型的权重值，所有元素通过这个权重来进行从小到大的排序。元素不可以重复，但是权重是可以重复的。一个Zset 类型的键可以存储 $2^{32} - 1$ 个元素。

Zset 的底层实现分为两种：

  • ziplist：压缩列表，在Zset 的元素个数小于128，每个元素的值小于64字节时，使用 ziplist；
  • skiplist：跳表，在不满足使用 ziplist 的条件时，使用 skiplist。

在Redis 7.0版本之后，ziplist 被弃置，使用 listpack 结构（紧凑列表）来代替。

底层数据结构转换的阈值可以通过修改配置来进行设置：

  • zset-max-listpack-entries： 用于配置元素个数，默认128；
  • zset-max-listpack-value： 用于配置元素值的大小，默认64。

skiplist

跳表是一种随机化的数据结构，本质是一种可以进行二分查找的有序链表。跳表在原有的有序链表上增加了多级索引，通过索引来实现快速查找。跳表不仅能提高搜索性能，也可以提高插入和删除操作的性能。

属性 类型 作用
header zskiplistNode* 指向头节点
tail zskiplistNode* 指向尾节点
length unsigned long 记录数据数量
leven int 记录最大层数
属性 类型 作用
obj SDS 记录实际数据
score double 记录权重值
backward zskiplistNode* 记录上一个节点地址
level[] zskiplistLevel 是一个数组，记录之后多个节点的地址和跨度
为什么使用skiplist，而不是用平衡树呢？
  1. 内存方面：平衡树每个节点存放两个指针，而跳表中每个节点包含指针数目平均下来为 $\frac{1}{1 - p}$，内存占用更小。
  2. 操作方面：平衡树中，找到指定范围的小值之后，还需要使用中序遍历继续寻找其他不超过大值的节点，而跳表中范围查询很方便，仅仅需要找到最小值，然后通过链表进行遍历即可。
  3. 算法实现：跳表相对于平衡树的实现要简单得多，平衡树的插入和删除操作都可能引发子树的调整，逻辑复杂，但是跳表的插入和删除只需要修改相邻节点的指针即可，逻辑简单且高效。

使用场景

基本数据类型

  • String：缓存对象、常规计数、分布式锁、共享session信息等
  • List：消息队列
  • Hash：缓存对象、购物车等
  • Set：聚合计算（并集、交集、差集），例如点赞、共同关注、抽奖活动等
  • Zset：排序场景，例如排行榜、电话和姓名排序等

其他数据类型

  • BitMap：二值状态统计的场景，例如签到、判断用户登录状态、连续签到用户总数等
  • HyperLogLog：海量数据基数统计的场景，例如百万级网页UV计数等
  • GEO：存储地理位置信息的场景，例如打车服务
  • Stream：消息队列，相较于List 实现的消息队列，有两个特有的特性：可以自动生成全局唯一的消息ID，支持以消费组的形式消费数据

什么是消费组？
消费组指的是一组消费者，共同消费消息队列中的数据，每个消费组都有一个唯一的标识符，组内的消费者共同协作，确保每条消息只被组内的一个消费者处理一次。

消费组的工作原理

  1. 消息分配
    • 消息队列中的每条消息通常都会有一个唯一的偏移量(Offset)。
    • 消费组内的消费者相互协作，分配各自要处理的消息范围。
  2. 消息消费
    • 一个消费者处理消息之后，会向系统发送确认，表示该消息已经被处理。
    • 系统会记录每个消费者的消费进度，以便在消费者重新启动时能够恢复消息进度。
  3. 负载均衡
    • 如果消费组内消费者数量发生变化，系统会重新分配消息给各个消费者，保证负载均衡和高可用性。

消费组的优点

  1. 并行处理：消费组允许多个消费者并行处理消息，提高了系统的吞吐量。
  2. 高可用性：如果消费组内的某个消费者发生故障，其他消费者可以接管未处理的消息，保证系统的高可用性。
  3. 负载均衡：系统会自动进行负载均衡，确保各个消费者负载大致分布均匀。
